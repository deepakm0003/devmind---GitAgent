# Detect Drift

## Purpose

Every time devmind writes code, it checks its own work. This is not optional and not skippable. Drift happens gradually — a new import here, a slightly different naming pattern there — until the codebase sounds like it was written by three different teams. Detect-drift is the circuit breaker.

This skill runs automatically after `write-feature`. It can also be invoked manually at any time to audit existing files.

## Input

This skill receives from the calling skill (`write-feature`):
- A list of files written in the current session (paths only)
- The current `memory/MEMORY.md`
- The current `knowledge/project-decisions.md`

If invoked manually (not from write-feature), ask the developer: `"Which files should I audit for drift? Or say 'all' to scan the full project."`

## Step 1: Load Constraint Sources

Read and extract from `memory/MEMORY.md`:
- `### Anti-Patterns (never do these)` — full section
- `### Architecture Decisions` — full section
- `### Conscious Exceptions` — if it exists, note what was already logged as intentional so we don't re-flag it

Read and extract from `knowledge/project-decisions.md`:
- `## Patterns Explicitly Avoided` — full section
- `## Naming Conventions` — full section
- `## Tech Stack (detected)` — the dependencies list

## Step 2: Run Git Diff

For each newly written file, run:
```bash
git diff HEAD -- [filepath]
```

If the file is new (not yet committed), run:
```bash
git diff --cached -- [filepath]
```

If neither produces output (file not staged), read the file directly.

Collect the full diff or file content for analysis.

## Step 3: Analyze for Drift

For each file, check the following. Each check is independent:

### Check A: Naming Convention Violations
Compare function names, variable names, file names against `knowledge/project-decisions.md → Naming Conventions`.
Flag if: a new function uses snake_case in a camelCase project, a new file uses PascalCase in a kebab-case project, etc.

### Check B: New Imports Not in Package Manifest
Extract all import statements from the diff. Cross-reference against:
- `package.json dependencies + devDependencies` (Node)
- `requirements.txt` or `pyproject.toml` (Python)
- `go.mod` (Go)

Flag any import that is not present in the manifest AND is not a relative project import.

### Check C: Anti-Pattern Violations
For each anti-pattern in `MEMORY.md → Anti-Patterns`:
- Parse the anti-pattern into a searchable concept (e.g., "global state" → look for `global`, `window.`, module-level mutable variables)
- Check if the diff introduces this pattern

Do NOT flag false positives. Only flag if you can point to a specific line.

### Check D: Patterns Explicitly Avoided
For each entry in `knowledge/project-decisions.md → Patterns Explicitly Avoided`:
- Convert to a code search pattern
- Check if the diff introduces this pattern

### Check E: Conscious Exception Cross-Reference
Before reporting any finding, check if it was already logged as a `## Conscious Exception` in MEMORY.md. If yes, skip — this was intentional.

## Step 4: Output

### If NO drift found:

Append to `memory/MEMORY.md` under `## Memory Log`:
```
- ✅ Drift check [ISO date]: clean — [N] files checked, [N] constraints verified
```

Output to developer: `"✅ Drift check: clean. [N] files match your established patterns."`

---

### If drift found:

For each drift item, create a structured report.

**First**: Create or append to `knowledge/drift-log.md`:

```
## Drift Report — [ISO date]
Generated by: devmind detect-drift after write-feature session

### Drift Item [N]
**File:** [filename, line range]
**Type:** [Naming Convention / New Dependency / Anti-Pattern / Avoided Pattern]
**What was written:**
```[language]
[specific line(s) from diff that triggered the flag]
```
**Constraint violated:** "[exact quote from MEMORY.md or project-decisions.md]"
**Logged on:** [date of that memory entry]
**Recommendation:** [specific, actionable fix — e.g., "Rename `getUserData` to `get_user_data` to match snake_case convention"]

---
```

**Second**: Create a git commit with the drift report:
```
⚠️ devmind: drift detected — [short description of main drift type]
```

**Third**: Output immediately to developer:

```
⚠️ Drift detected in [N] file(s).

[For each drift item:]
In [filename] (line [N]):
  You wrote: [code snippet]
  But your memory says: "[constraint quote]"
  Fix: [recommendation]

Full report: knowledge/drift-log.md
What would you like to do?
1. Fix it now — I'll revise the affected files
2. Log it as a conscious exception — tell me why this is intentional
3. Update the convention — tell me the new rule
```

Wait for developer response before taking any action.

## Step 5: Handle Response

- **Fix it now**: Invoke `write-feature` targeting only the flagged sections. Run detect-drift again after revision. Do not stop until drift check is clean or all remaining items are conscious exceptions.
- **Log as exception**: Ask for reasoning, append to `memory/MEMORY.md → Conscious Exceptions`, output confirmation.
- **Update convention**: Update `knowledge/project-decisions.md → Naming Conventions` (or relevant section) with new rule. Archive old version. Output confirmation.
