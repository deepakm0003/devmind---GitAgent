# Build Mental Model

## Purpose

This skill performs a structural analysis of the project's files — without asking the developer anything. It observes naming conventions, file organization, tech stack, testing patterns, and recent activity. Its output is `knowledge/project-decisions.md`, which becomes the second source of truth (alongside MEMORY.md) that all builder skills depend on.

Run this skill after `interview-project` completes, or independently when the developer wants devmind to re-scan after major refactoring.

## Step 1: Detect Package Manager and Tech Stack

Check for the following files in order. Use the first one found:

1. `package.json` — Node.js / JavaScript / TypeScript project
2. `requirements.txt` or `pyproject.toml` or `setup.py` — Python project
3. `go.mod` — Go project
4. `Cargo.toml` — Rust project
5. `pom.xml` or `build.gradle` — Java/JVM project
6. `composer.json` — PHP project
7. `Gemfile` — Ruby project

Read the detected file and extract: runtime/language, key dependencies, and any scripts/commands defined.

If none of the above are found, note: "Tech stack: undetected — no recognized package manifest found."

## Step 2: Read Directory Structure

Run: `find . -maxdepth 3 -not -path '*/node_modules/*' -not -path '*/.git/*' -not -path '*/dist/*' -not -path '*/__pycache__/*' -not -path '*/vendor/*'`

(On Windows, use: `Get-ChildItem -Recurse -Depth 3 | Where-Object { $_.FullName -notmatch 'node_modules|\.git|dist|__pycache__|vendor' } | Select-Object FullName`)

Analyze the output for:
- Top-level folder naming conventions (is it `src/`, `lib/`, `app/`, `pkg/`?)
- Separation patterns (is there a `tests/` or `__tests__` or `*.spec.ts` co-location pattern?)
- Configuration file locations
- Whether there's a monorepo structure (multiple package.json / go.mod files)

## Step 3: Read Source Files

Identify the 3–5 files most likely to represent the project's core patterns. Prioritize:
- The largest files by line count in the primary source directory
- Files modified most recently (run: `git log --name-only --format="" -20 | sort | uniq -c | sort -rn | head -10` to find most-touched files)
- Entry points (e.g., `index.ts`, `main.go`, `app.py`, `server.js`)

Read each file. Observe:
- Naming conventions: camelCase, snake_case, PascalCase, kebab-case — for files, functions, variables, types
- Comment style: JSDoc, inline, docstrings, or none
- Error handling pattern: exceptions, error returns, Result types, callbacks
- Import/module style: ES modules, CommonJS, relative imports, path aliases
- Whether types are used and how strictly

## Step 4: Read Existing MEMORY.md

Read `memory/MEMORY.md` if it exists. Extract:
- Any explicitly stated anti-patterns
- Any architecture decisions
- Any failed experiments that may indicate patterns to avoid

Cross-reference: if the code scan observes something that contradicts MEMORY.md, flag it — do not silently accept either. Write a note under "Observed Contradictions" in project-decisions.md.

## Step 5: Check for Tests

Look for test files using patterns: `*.test.*`, `*.spec.*`, `_test.go`, `test_*.py`, files inside `tests/`, `__tests__/`, `spec/`.

If tests exist:
- Note the testing framework (Jest, Pytest, Go test, RSpec, etc.)
- Note whether tests co-locate with source or live in a separate directory
- Note the assertion style
- Note whether mocks/stubs exist and how they're structured

If no tests exist: note "No test suite detected. New code should not introduce a testing framework unless explicitly approved by developer."

## Step 6: Write knowledge/project-decisions.md

Create or overwrite `knowledge/project-decisions.md` using this exact structure:

```
# Project Decisions
> Auto-generated by devmind build-mental-model skill. Last updated: [ISO date]
> Do not edit manually — re-run build-mental-model to refresh.

## Tech Stack (detected)
- Language/Runtime: [detected]
- Key dependencies: [list top 5-10 from package file]
- Build/run commands: [from scripts section]

## Naming Conventions (observed from code)
- Files: [convention observed]
- Functions/methods: [convention observed]
- Variables: [convention observed]
- Types/Classes: [convention observed]
- Constants: [convention observed]
- Test files: [convention observed, or "no tests found"]

## File Organization Pattern
[Describe the folder structure pattern in 2-4 sentences. Is it feature-based? Layer-based? Flat?]

## Testing Approach
[Describe the testing setup, framework, and style. Or: "No test suite detected."]

## Current Work-In-Progress (inferred from recent files)
[Based on recently modified files from git log, describe what appears to be actively in development]

## Patterns Explicitly Used
[List patterns observed in the code: e.g., repository pattern, middleware chain, event emitters, etc.]

## Patterns Explicitly Avoided (from MEMORY.md)
[List patterns from MEMORY.md Anti-Patterns section. If MEMORY.md not yet present: "Interview not yet complete."]

## Observed Contradictions
[Any cases where scanned code contradicts MEMORY.md. If none: "None detected."]
```

## Step 7: Update MEMORY.md

Append to the `## Memory Log` section of `memory/MEMORY.md`:

```
- [ISO date] Mental model built. Tech stack: [detected]. [N] files scanned. [N] contradictions flagged.
```

If MEMORY.md does not yet exist (build-mental-model was run before interview-project), create a minimal MEMORY.md with:
```
# devmind Memory
> Interview not yet complete. Mental model built from file scan only.
> Last updated: [ISO date]

## Memory Log
- [ISO date] Mental model built without interview. Run interview-project to complete onboarding.
```

## Completion Output

Output to developer:

`"Mental model complete. I've scanned [N] files and documented your stack, conventions, and patterns in knowledge/project-decisions.md. [If contradictions found: 'I also found [N] places where the code contradicts your stated memory — check the Observed Contradictions section.'] Ready to build."`
